# Session – 2026-04-14

### Topics covered
* **Cardinality:** The difference between High Cardinality (many unique values, e.g., emails) and Low Cardinality (few unique values, e.g., status flags or `site_id` 1–5).
* **Database Scan Types:** When the optimizer chooses a **Full Table Scan** versus an **Index Range Scan** or **Index Seek**.
* **Index Architecture:** How **B-Tree** indexes work (sorting, balancing, ranges) versus **Hash** indexes (exact matches only, no ranges).
* **Composite Indexes:** Multi-column indexes and the critical importance of the **Left-Prefix Rule** (Leading Column rule).
* **SARGability:** How wrapping columns in functions (like `TO_CHAR` or `TRUNC`) breaks the database's ability to use an index.
* **Performance Trade-offs:** The concept of **DML Overhead**—how indexes speed up reads (`SELECT`) but slow down writes (`INSERT`, `UPDATE`, `DELETE`).

## What I understood
* **Set Operators stack rows:** Unlike `JOIN`s, which combine columns from different tables side-by-side, Set Operators stack the resulting rows of different queries vertically.
* **UNION vs. UNION ALL:** `UNION` merges the results of two queries into one list, but it does extra work to remove any duplicate rows.  `UNION ALL` merges them and simply keeps everything (including duplicates), which makes it run faster. 
* **MINUS (or EXCEPT):** This acts like data subtraction.  It takes the complete results from the top query and removes any rows that also happen to appear in the bottom query. 
* **INTERSECT:** This finds the exact overlap.  It will only return a row if it successfully popped up in the results of *both* the first and second query.
* **The Golden Rule:** To link queries with these operators, both `SELECT` statements must ask for the exact same *number* of columns, and those columns must have matching (or compatible) data types in the exact same order.
